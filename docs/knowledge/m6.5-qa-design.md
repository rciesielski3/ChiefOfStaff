# M6.5 Evergreen Knowledge Base — QA Component Design

**Date:** 2026-07-21  
**Status:** Design complete, ready for M6.5 sprint  
**Scope:** KnowledgeQualityValidator service (evolved from validate-knowledge-extraction.ts CLI)  
**Owner:** M6.5 implementation phase

---

## Executive Summary

The evergreen knowledge base (M6.5) requires quality assurance before facts enter permanent storage. This spec defines the QA component architecture, quality gates, and reporting that will ensure knowledge base integrity while tracking quality metrics over time.

This component repurposes the existing validate-knowledge-extraction.ts CLI tool into a production-grade QA service with both programmatic and CLI interfaces.

### Service Interface

```typescript
export interface ValidationResult {
  passed: Fact[];
  rejected: Fact[];
  report: QAReport;
}

export interface QAReport {
  timestamp: string;
  facts_validated: number;
  facts_passed: number;
  facts_rejected: number;
  rejections_by_reason: { [reason: string]: number };
  confidence_stats: {
    min: number;
    max: number;
    mean: number;
    median: number;
  };
  sensitivity_distribution: {
    PUBLIC: number;
    PRIVATE: number;
    UNCERTAIN: number;
  };
}

export class KnowledgeQualityValidator {
  validateBatch(facts: Fact[]): ValidationResult
  validateFact(fact: Fact): { valid: boolean; reasons: string[] }
  generateReport(result: ValidationResult): string  // Table or JSON
}
```

### CLI Interface (Keep & Enhance)


## Quality Gates (M6.5 Will Enforce)

### Gate 1: Confidence Threshold

| Rule | Value | Action |
|------|-------|--------|
| Minimum confidence | ≥0.5 | Reject if below |
| Rationale | Facts with <0.5 confidence are unreliable | Prevent low-signal facts |

### Gate 2: Structural Validation

| Field | Required? | Action |
|-------|-----------|--------|
| `id` | Yes | Reject if missing |
| `content` | Yes | Reject if empty |
| `type` | Yes | Reject if invalid type |
| `confidence` | Yes | Reject if NaN/invalid |
| `sensitivity` | Yes | Reject if not PUBLIC/PRIVATE/UNCERTAIN |

### Gate 3: Duplicate Detection

| Check | Scope | Action |
|-------|-------|--------|
| ID uniqueness | Within batch | Reject duplicates |
| Content hash | Against existing KB | Flag (handled by evolution layer) |

### Gate 4: Sensitivity Accuracy

| Scenario | Check | Action |
|----------|-------|--------|
| PUBLIC fact with PII | Flag mismatch | Log as quality issue |
| PRIVATE fact no PII | Spot-check | Count as over-cautious (acceptable) |
| UNCERTAIN facts | Accept as-is | Mark for manual review |

**Rationale:** Zero false negatives on PII (safety-critical), acceptable false positives (conservative)

## Notes

This design is intentionally conservative:
- **Quality gates are strict** (confidence ≥0.5, structural validation) to ensure KB reliability
- **Reporting is comprehensive** (weekly, trackable, archived) for observability
- **False positives acceptable** (UNCERTAIN sensitivity is fine, easier to review than miss PII)
- **Pragmatic scope** (reuses existing CLI tool, minimal refactoring, clear integration path)

The component is designed to scale from 100 facts (M6.3 launch) to 100K+ facts (mature KB) while maintaining quality assurance at every stage.
